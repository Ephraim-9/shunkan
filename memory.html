🤖 PROTOCOL: How Future AI Agents Must Iterate on This Project

Any AI coding assistant or subagent invoked to work on Project Shunkan must adhere strictly to the following execution lifecycle:

  1. Read Before Writing: Before modifying code in `core/`, `desktop-linux/`, or `mobile-android/`, verify invariant compliance by checking this page (`memory.html`) and `stack.html`.
  2. Update `tasks.html` Synchronously: When commencing a task (e.g., `TASK-102`), verify requirements. Once automated tests pass (`Verification Command`), you must directly edit `tasks.html` source code to mark the item as completed (`status-completed`) and update the input checkbox attribute (`checked`).
  3. Document Architectural Deviations (`ADR`): If OS updates (e.g., Android 16 or Wayland protocols) force a modification to the network or clipboard handling logic, create a new Architectural Decision Record in the table below before writing code.

🛡️ Absolute Architectural Invariants

Non-negotiable constraints that must never be broken:

  • INV-01 (Memory Cap): The Linux system tray daemon (desktop-linux) targets ~15MB of idle memory as its operating baseline and must never exceed 20 MB as its hard ceiling. Do not introduce Node.js runtime engines or Chromium webview background loops outside of the explicit Tauri UI overlay window.
  • INV-02 (Zero Cloud Reliance): All device pairing, mDNS discovery (_shunkan-sync._udp.local., port 4433/udp), and QUIC socket transmissions must occur strictly over local IP/subnets. Do not hardcode external STUN, TURN, or telemetry server URLs.
  • INV-03 (No Polling on Android): Never write while(true) or Handler.postDelayed() loops on Android main/service threads to poll clipboard states. Rely strictly on TileService, ShareTarget, and ShunkanIME (InputMethodService) event hooks. Never use AccessibilityService for clipboard capture — prohibited by Play Store policy.
  • INV-04 (FFI Type Safety): All bridge interfaces between shunkan-core and Kotlin must be defined using uniffi interface definitions (.udl or proc-macros). Never write raw JNI C-style byte manipulation boilerplate.
  • INV-05 (BLAKE3 for File Chunks): All chunked file integrity verification in shunkan-core must use BLAKE3 (blake3 crate). SHA-256 is prohibited for file streaming — it bottlenecks CPU throughput at Wi-Fi 6 speeds above 100 MB/s. BLAKE3's SIMD throughput is 15–20 GB/s.

⚠️ Known Edge Cases & Mitigation Strategies

Tricky scenarios documented during OS feasibility research:

Linux Firewall Blocking UDP 5353 (mDNS) & 4433 (QUIC) Mitigated
If ufw or firewalld blocks multicast UDP port 5353 (mDNS) or the QUIC default port 4433, discovery and transport fail. Mitigation: Our CLI/GUI displays a one-click notification advising the exact rule: sudo ufw allow 5353/udp && sudo ufw allow 4433/udp. Service record is advertised as _shunkan-sync._udp.local. (pure UDP, RFC 6763 compliant).
Android Aggressive Doze / Battery Saver Mitigated
OEM battery managers (Xiaomi MIUI, Samsung OneUI) kill background sockets. Mitigation: Socket reconnects instantly via TileService tap or ShareSheet activation (0-RTT QUIC resume).
X11 Clipboard Feedback Loop Deadlock Solved
When shunkan-core writes remote text into Linux X11 clipboard, arboard might re-trigger a local copy event, looping infinitely. Mitigation: Tag clipboard payloads with an internal hash nonce; skip broadcast if nonce matches last received peer payload.
GNOME Wayland: Synthetic Key Injection Blocked Mitigated
GNOME/Mutter blocks virtual-keyboard-v1 protocol for security, preventing wtype from injecting Ctrl+V keystrokes into other windows. Mitigation: On GNOME Wayland sessions, Shunkan writes the selected history item directly to the Wayland clipboard and displays a toast notification prompting the user to paste with Ctrl+V. On wlroots compositors (Hyprland/Sway/KDE), full auto-paste via wtype (virtual-keyboard-v1) is supported.

📜 Architectural Decision Records (ADR)

ADR-001: Selection of QUIC (quinn) over WebSockets/TCP for P2P Transport Approved • July 2026
Context: We required a transport protocol capable of simultaneously transmitting tiny clipboard strings (< 100 bytes) with zero latency while transferring multi-gigabyte files across Wi-Fi without blocking.
Decision: Adopted quinn crate implementing QUIC over UDP (port 4433/udp). No TCP fallback in v1.0 — TCP is explicitly excluded per this decision.
Consequences: Eliminates head-of-line blocking via independent stream multiplexing (Stream 0 for files, Stream 1 for clipboard events). Requires UDP port opening on Linux desktop firewalls.
ADR-002: Adoption of UniFFI for Android Kotlin Bindings Approved • July 2026
Context: Calling Rust code from Kotlin/JVM historically required fragile, manual JNI (Java Native Interface) C bridges prone to segmentation faults and memory leaks.
Decision: Use Mozilla's uniffi crate to auto-generate type-safe Kotlin classes from Rust interface definitions.
Consequences: Reduces boilerplate across mobile-android/ by 90%. Guarantees thread safety when passing asynchronous streams and complex data structs across the JNI boundary.
ADR-003: BLAKE3 (blake3 crate) for File Chunk Integrity Verification Approved • July 2026
Context: File transfer targets Wi-Fi 6 speeds >100 MB/s. SHA-256 hashing of 64 KB chunks is CPU-bound and becomes a bottleneck on ARM mobile processors at these speeds.
Decision: Replace SHA-256 with BLAKE3 (blake3 crate) for all file chunk integrity verification and deduplication nonces in shunkan-core.
Consequences: BLAKE3 achieves 15–20 GB/s throughput via SIMD parallelism and native Merkle-tree chunking — orders of magnitude faster than SHA-256. Natively supports incremental verification for resumable transfers. Standard in modern Rust P2P engines (e.g., iroh).
ADR-004: Tiered Wayland Clipboard Protocol Strategy Approved • July 2026
Context: org.freedesktop.portal.Clipboard (XDG Desktop Portal) is only accessible inside an active org.freedesktop.portal.RemoteDesktop session and is not available for standalone daemon clipboard monitoring on GNOME Wayland. Additionally, GNOME/Mutter blocks the virtual-keyboard-v1 Wayland protocol, preventing wtype from injecting keystrokes.
Decision: Use a tiered compositor-aware strategy in clipboard.rs:
 1. wlroots (Hyprland/Sway/KDE): ext-data-control-v1 via wl-clipboard-rs for monitoring; wtype for injection.
 2. GNOME Wayland: wl-paste --watch-equivalent loop via wl-clipboard-rs + xdg-desktop-portal-gnome auth; clipboard-write fallback + toast for paste injection.
 3. X11: arboard polling + enigo (Xtest) injection.
Consequences: Full auto-paste works on X11 and wlroots compositors. GNOME Wayland requires manual Ctrl+V from user after clipboard history selection (security trade-off).
