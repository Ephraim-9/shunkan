Product Requirements Document (PRD)

An ultra-low latency, decentralized, cross-platform clipboard history and file-sharing ecosystem designed for Linux and Android with native UI replacement.

⚡ Executive Summary

Project Shunkan (瞬間) is a local-first, decentralized peer-to-peer (P2P) synchronization engine designed to seamlessly link Linux desktops and Android mobile devices across any permutation (Linux $\leftrightarrow$ Android, Linux $\leftrightarrow$ Linux, Android $\leftrightarrow$ Android).

Unlike traditional cloud-dependent clipboards or clunky web-based file senders, Shunkan operates over **local Wi-Fi/LAN** using zero-configuration mDNS discovery and encrypted QUIC/TLS 1.3 streams. It aims to not only sync copied items instantly (瞬間) but to **completely replace the native OS clipboard history UI** with a unified, instant, searchable command palette on Linux and a quick-action share/overlay system on Android.

🎯 Core Objectives

  • Zero-Config Discovery: Instant local network pairing via mDNS. No IPs to type.
  • Native UI Replacement: Replace standard OS clipboard popups with our advanced, searchable history UI.
  • Sub-50ms Latency: Text clipboard changes propagate across devices in milliseconds.
  • Gigabit File Transfers: Multi-gigabyte file streaming over TCP/QUIC utilizing full Wi-Fi 6 bandwidth.
  • 100% Local & Encrypted: E2E encryption via ephemeral curve exchange. No data touches the cloud.

🧑‍💻 Target User Personas & Use Cases

The Power Developer (Linux + Android)

Copies a complex API key or bash script snippet on their Linux terminal (`Ctrl+Shift+C`) and instantly taps `Paste` on their Android phone inside Termux or a testing environment without wires or email drafts.

The Multi-Device Designer

Takes a screenshot or records a 2GB screen capture on their Android phone, opens the system Share Sheet, selects "Shunkan Sync," and has the file ready on their Linux desktop downloads folder within 3 seconds.

The Privacy Enthusiast

Demands that sensitive OTP codes, passwords, and personal photos never leave the local LAN router or pass through external telemetry servers (like Google Drive or Telegram saved messages).

🔒 Non-Functional Requirements

  • Memory Efficiency: The background daemon on Linux must consume less than 20 MB of RAM when idle (achieved via pure Rust / GTK / Tauri).
  • Battery Optimization: On Android, network sockets must use adaptive polling or wake-locks only when a peer is actively transmitting, keeping background battery drain < 1% per 24 hours.
  • Wayland & X11 Parity: Linux core must auto-detect `$XDG_SESSION_TYPE` and dynamically select either Xlib/XCB bindings or `wlr-data-control` / XDG Desktop Portal without crashing.
  • Security & Sandbox: Must reject unauthenticated connections automatically and prevent directory traversal (`../../`) during file saving.

🚫 Out of Scope (v1.0)

  • External Cloud Relay: No STUN/TURN servers for syncing over the internet (4G/5G across cities) in v1.0. Must be on the same local network / VPN (WireGuard/Tailscale compatible).
  • iOS & Windows Desktop Clients: While the core Rust protocol (`quinn` + mDNS) will cleanly compile to Windows (`.exe`) and macOS/iOS in the future, v1.0 strictly targets Linux (`.deb`/`.AppImage`/`Flatpak`) and Android (`.apk`).
  • Shared Screen / Remote Desktop: VNC or display mirroring is not included; focus is strictly on clipboard history and file pipelines.

🤖 AI Agent Architectural Invariants

🧠

Directives for Future AI Agents Parsing This PRD

Whenever tasked with modifying or extending this product, AI agents MUST preserve two core tenets:
1. Never introduce polling loops on Android main threads to bypass clipboard blocks; always utilize the event-driven `TileService`, `ShareSheet`, or `IME` hooks detailed in `research.html`.
2. Do not use bloated web frameworks (Electron) for the Linux desktop daemon. The core logic must reside in `shunkan-core` with high-speed async execution to maintain the < 20MB RAM non-functional constraint.
