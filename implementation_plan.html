Complete Implementation Plan

A phased, end-to-end blueprint covering the entire lifecycle from core Rust library development through CI/CD pipelines and multi-format packaging.

🦀 Phase 1: Core Rust Library (shunkan-core)
Foundation

Build the shared P2P engine as a pure Rust crate. All networking, cryptography, protocol, and history logic lives here — zero platform-specific UI code.

1.1 — Discovery Module (discovery.rs)

mDNS-SD Broadcasting & Listening

Integrate mdns-sd and local-ip-address crates. Advertise _shunkan-sync._udp.local. service records containing hostname, OS type, and QUIC port (4433/udp). Build an async background listener that emits PeerDiscovered / PeerLost events via tokio::sync::broadcast.

1.2 — Transport Module (transport.rs)

QUIC Server & Client Endpoints

Setup quinn server and client endpoints with rustls self-signed TLS 1.3 certificates generated via rcgen. Implement multiplexed stream management — Stream 0 for file transfers, Stream 1+ for lightweight clipboard events. Support 0-RTT reconnection for paired devices.

1.3 — Protocol Module (protocol.rs)

Message Frame Definitions

Define typed message frames using serde + bincode: ClipboardItem (text/rich-text/small-image), FileChunkHeader, FileChunkData, HandshakeRequest, HandshakeResponse, PeerStatus. All frames include a 4-byte length prefix and a frame-type discriminator byte.

1.4 — Crypto Module (crypto.rs)

X25519 Key Exchange & PIN Verification

Implement Diffie-Hellman ephemeral key exchange using x25519-dalek. Generate 6-digit PIN codes for first-time pairing. Store mutual peer public keys and TLS certificates in local persistent storage (directories crate for platform-appropriate paths).

1.5 — Hashing (transport.rs integration)

BLAKE3 Integrity Verification

Use the blake3 crate for all file chunk integrity verification. Per INV-05, SHA-256 is prohibited — BLAKE3 achieves 15–20 GB/s via SIMD. Implement rolling hash nonces for clipboard deduplication to prevent feedback loops.

1.6 — History Module (history.rs)

SQLite LRU Clipboard History

Integrate rusqlite for persistent clipboard history with LRU eviction (default: 500 items). Support full-text search via SQLite FTS5. BLAKE3 hash deduplication prevents storing identical items.

🐧 Phase 2: Desktop Linux Application
Desktop UI

Build the Linux desktop frontend using Tauri v2 — a lightweight system tray daemon with a glassmorphic clipboard history command palette.

2.1 — Tauri v2 Daemon & System Tray

Background Daemon Bootstrap

Create Tauri v2 project under desktop-linux/. Configure tauri-plugin-system-tray to run headless on boot, display connection status in the system tray, and embed shunkan-core as a background Tokio runtime thread. Target: <15MB idle RAM (hard ceiling: <20MB per INV-01).

2.2 — Multi-Session Clipboard Adapter

Wayland + X11 Clipboard Listener

Auto-detect $XDG_SESSION_TYPE at startup. X11: Use arboard polling at ~100ms for CLIPBOARD and PRIMARY atoms. Wayland (wlroots): Use wl-clipboard-rs via ext-data-control-v1. GNOME Wayland: wl-paste --watch-equivalent loop + xdg-desktop-portal-gnome authorization.

2.3 — Glassmorphic Command Palette UI

Hotkey-Activated History Popup

Register Super + V via tauri-plugin-global-shortcut. Display a floating, searchable clipboard history window with glassmorphic CSS. On item selection: X11 — inject Ctrl+V via enigo (Xtest); wlroots — use wtype (virtual-keyboard-v1); GNOME — clipboard-write + toast notification.

📱 Phase 3: Android Mobile Application
Mobile UI

Build a native Kotlin + Jetpack Compose Android app that embeds libshunkan_core.so via UniFFI, with system-level OS hooks for seamless clipboard and file sharing.

3.1 — Android Studio Project Setup

NDK Integration & UniFFI Bindings

Configure build.gradle.kts with NDK support. Cross-compile libshunkan_core.so for arm64-v8a and x86_64 targets using cargo-ndk. Integrate auto-generated UniFFI Kotlin bindings into com.shunkan.sync.core.

3.2 — Quick Settings TileService

One-Tap Clipboard Sync

Implement ShunkanTileService extending android.service.quicksettings.TileService. When tapped, fetch the latest clipboard item from paired Linux devices or push the current Android clipboard to Linux — all within the privileged tile context.

3.3 — ShareActivity & Direct Share Target

System Share Sheet Integration

Register ACTION_SEND / ACTION_SEND_MULTIPLE intent filters. Implement ShortcutManager.pushDynamicShortcut() so paired Linux devices appear in the Direct Share top row. Stream shared text/files immediately over QUIC.

3.4 — Optional Custom Keyboard (ShunkanIME)

True Background Auto-Capture

Implement lightweight InputMethodService with ClipboardManager.OnPrimaryClipChangedListener. For power users: optional Shizuku privilege elevation (appops set com.shunkan.sync READ_CLIPBOARD allow). Never use AccessibilityService — prohibited by Play Store policy.

📦 Phase 4: File Streaming Engine
Performance

Implement high-speed chunked file transfer over QUIC with BLAKE3 integrity verification.

4.1 — Chunked QUIC Transfer Protocol

64 KB Chunk Streaming with BLAKE3

Stream binary files in 64 KB chunks over dedicated QUIC streams. Each chunk includes a BLAKE3 rolling hash for real-time integrity verification. Implement Merkle-tree chunked verification for resumable integrity checks. Broadcast transfer progress events to UI layers.

4.2 — Auto-Resume & Error Recovery

Resumable Transfers

If Wi-Fi drops mid-transfer, persist the last acknowledged chunk offset. On reconnection (QUIC 0-RTT), resume from the exact byte offset. Save received files to $HOME/Downloads/Shunkan on Linux and Documents/Shunkan on Android.

4.3 — Drag-and-Drop & File Picker Integration

Platform-Native File Selection

Linux: Tauri drag-and-drop event handlers. Android: Intent.ACTION_OPEN_DOCUMENT file picker and Share Sheet file receiver. Display real-time progress bars with speed metrics (MB/s).

🧪 Phase 5: Testing
Quality

Comprehensive test coverage across all modules and platforms. See testing.html for the full test plan.

5.1 — Unit Tests

Core Module Unit Tests

Unit tests for each shunkan-core module: discovery.rs, transport.rs, protocol.rs, crypto.rs, history.rs. Target: >80% code coverage. Run via cargo test -p shunkan-core.

5.2 — Integration Tests

Cross-Module & Network Tests

Integration tests combining discovery → handshake → clipboard sync → file transfer in a single test harness. Uses loopback interfaces (127.0.0.1) for deterministic network testing.

5.3 — Platform Tests

Desktop & Mobile Verification

Desktop Linux clipboard adapter tests under both X11 and Wayland sessions. Android instrumented tests via ./gradlew connectedAndroidTest for TileService, ShareActivity, and UniFFI binding validation.

🔄 Phase 6: CI/CD Pipeline
Automation

GitHub Actions workflow for continuous integration, testing, and release artifact building.

6.1 — GitHub Actions CI

Automated Lint, Test & Build

Trigger on push/PR to main. Jobs: lint (cargo clippy -- -D warnings), test (cargo test --workspace), build-linux (cargo build --release). Cache ~/.cargo/registry and target/ directory for fast CI runs. See .github/workflows/ci.yml.

6.2 — Android CI (Planned)

Gradle Build & Unit Tests

Future job: Setup Android SDK/NDK, cross-compile libshunkan_core.so for ARM64, run ./gradlew assembleDebug and ./gradlew testDebugUnitTest. Will be added when Phase 3 implementation begins.

📤 Phase 7: Packaging & Distribution
Release

Multi-format packaging for Linux distributions and Android devices.

7.1 — Linux Packaging

.deb, .AppImage, Flatpak

.deb: Use cargo-deb to generate Debian packages with systemd service unit for auto-start. .AppImage: Bundle via appimagetool for universal Linux distribution. Flatpak: Create com.shunkan.sync.flatpak manifest with portal permissions for clipboard and network access.

7.2 — Android APK

Signed Release APK

Build signed release APK via ./gradlew assembleRelease. Include libshunkan_core.so for both arm64-v8a and x86_64 ABIs. Target minimum SDK: API 29 (Android 10).

7.3 — GitHub Releases

Automated Release Publishing

Tag-triggered CI job that builds all artifacts (.deb, .AppImage, .apk) and uploads them to GitHub Releases with checksums and changelogs.
