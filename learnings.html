Learnings, Errors & Hard-Won Knowledge

A living record of technical pitfalls, crate compatibility issues, and architectural lessons encountered during Project Shunkan development. Future AI agents must consult this before starting work.

📦 Crate Version Compatibility Issues

The Rust ecosystem evolves rapidly — pinning compatible crate versions across the quinn + rustls + rcgen dependency chain is critical to avoid compile failures.

quinn + rustls Version Matrix Critical

quinn depends on specific rustls major versions. Mixing quinn 0.11.x with rustls 0.23.x may cause trait incompatibility errors because quinn re-exports its own rustls types.

⚠️ Error Pattern

error[E0308]: mismatched types — expected rustls::ServerConfig, found rustls::ServerConfig
This occurs when two different versions of rustls are compiled — one from quinn's dependency tree and one from your direct Cargo.toml.

✅ Resolution

Always use the rustls version that quinn re-exports. Check with cargo tree -i rustls and match your direct dependency. For quinn 0.11.x, use rustls = { version = "0.23", features = ["ring"] }. Import rustls types via quinn::rustls when possible.

rcgen Certificate Generation Compatibility Warning

rcgen is used to generate self-signed TLS certificates for QUIC endpoints. Newer versions of rcgen changed their API — Certificate::from_params() was replaced with CertificateParams::self_signed().

✅ Resolution

Pin rcgen to a version compatible with your rustls. For rustls 0.23.x with ring backend, use rcgen = "0.13". Always check rcgen's changelog for API migration guides when upgrading.

rustls CryptoProvider Initialization Warning

Starting with rustls 0.22+, you must explicitly install a crypto provider before creating TLS configs. Without this, ServerConfig::builder() panics at runtime.

// Required at startup before any TLS operations:
rustls::crypto::ring::default_provider()
    .install_default()
    .expect("Failed to install rustls crypto provider");

🏗️ Workspace Configuration Errors

Missing Cargo.toml for Workspace Members Critical

When defining a Cargo workspace with [workspace] members = ["core"] in the root Cargo.toml, the referenced directory must contain its own Cargo.toml with [package] metadata. A missing or empty member Cargo.toml causes:

error: failed to load manifest for workspace member `core`
Caused by: failed to read `/path/to/core/Cargo.toml`

✅ Resolution

Always create the member Cargo.toml with at minimum [package] name, version, and edition fields before adding it to the workspace. Use cargo init core --lib to scaffold correctly.

Virtual Workspace vs Package Workspace Info

A root Cargo.toml with only [workspace] (no [package]) is a "virtual workspace." You cannot run cargo build at the root without specifying -p <package> unless you add default-members. If the root also has a [package], it's a "package workspace" and the root itself is implicitly a member.

🔄 API Changes Between Crate Versions

quinn Endpoint API Evolution Info

The quinn::Endpoint API changed significantly between 0.10 and 0.11:

  • Endpoint::server() → Endpoint::new() with separate ServerConfig
  • Endpoint::connect() now requires explicit ClientConfig
  • Connection handling switched from NewConnection struct to direct Connection return
x25519-dalek API Changes Info

x25519-dalek 2.x renamed EphemeralSecret constructor from ::new() to ::random_from_rng() and requires explicit RNG parameter instead of relying on OsRng default.

🔍 mDNS Library Selection

mdns vs mdns-sd — Why We Chose mdns-sd Resolved

Two Rust mDNS crates exist with very different maturity levels:

  • mdns crate: Minimal, query-only — cannot advertise/register services. Only discovers existing mDNS records. Insufficient for Shunkan's bidirectional discovery requirement.
  • mdns-sd crate: Full DNS-SD implementation supporting both service registration (advertising) and browsing (discovery). Supports TXT records for metadata (hostname, OS type). Actively maintained with async support.

✅ Decision

Selected mdns-sd for bidirectional service discovery. Each Shunkan device both registers its own _shunkan-sync._udp.local. service record and browses for peers. The mdns crate would have required a second library for service registration.

📁 Directory Structure Restructuring

shunkan-core/ → core/ Rename Resolved

The initial plan used shunkan-core/ as the directory name for the shared Rust library. This was restructured to core/ for brevity and consistency with Rust workspace conventions.

  • Before: shunkan-core/Cargo.toml with name = "shunkan-core"
  • After: core/Cargo.toml with name = "shunkan-core"
  • Impact: All workspace member references, import paths, and documentation updated. The crate name in Cargo.toml remains shunkan-core for cargo test -p shunkan-core compatibility.
Workspace Root Cargo.toml Setup Info

The project uses a virtual workspace pattern where the root Cargo.toml defines workspace members but is not itself a package:

# /shunkan/Cargo.toml (workspace root)
[workspace]
members = ["core"]
resolver = "2"

This ensures cargo build --workspace compiles all members and shared dependencies are deduplicated across the lockfile.

💡 Additional Lessons Learned

Tokio Runtime in Library Crates Info

shunkan-core is a library crate — it must not include #[tokio::main] or create its own Tokio runtime. The runtime is owned by the application layer (Tauri on Linux, Android's coroutine scope via UniFFI). The core library exposes async fn that the caller drives.

BLAKE3 vs SHA-256 Performance Difference Resolved

Early prototyping confirmed that SHA-256 hashing of 64 KB chunks becomes CPU-bound at Wi-Fi 6 speeds (>100 MB/s). BLAKE3's SIMD-accelerated implementation (blake3 crate) achieves 15–20 GB/s — ensuring the hash computation is never the bottleneck during file transfers. This validated ADR-003.

Self-Signed Certificates & quinn Warning

QUIC (via quinn) requires valid TLS certificates. For P2P local-network use, self-signed certificates generated by rcgen are used. The client must configure ClientConfig to accept these by implementing a custom ServerCertVerifier or using danger_config in development. For production, implement proper certificate pinning using the exchanged peer public keys.

Firewall Rules for mDNS + QUIC Info

On Linux systems with ufw or firewalld enabled, both mDNS (UDP 5353) and QUIC (UDP 4433) must be explicitly allowed. Without this, peer discovery and transport silently fail with no error messages — making debugging extremely difficult.

# Required firewall rules:
sudo ufw allow 5353/udp   # mDNS multicast discovery
sudo ufw allow 4433/udp   # QUIC transport
